# 付费分析框架

本文件供 `game_monetization` skill 的 Phase 3 使用。在进行付费/变现深度分析时，按以下维度和术语保持一致性。

---

## 一、付费相关术语（与用户/报告用语对齐）

| 术语 | 含义 | 常见用户表述 |
|------|------|-------------|
| 付费率 / 付费转化率 | 付费用户占活跃用户的比例 | "多少人充了"、"转化" |
| ARPU | 每用户平均收入 | 总收入/DAU 或类似口径 |
| ARPPU | 每付费用户平均收入 | "大 R 均值"、"付费用户人均" |
| LTV | 用户生命周期价值 | LTV、生命周期价值 |
| 首充率 | 首次付费用户占比或首充转化 | "首充"、"第一次付费" |
| 首充时间 | 从注册到首次付费的时长 | "多久才充"、"首充时长" |
| 付费结构 | 收入来源构成（如皮肤/抽卡/通行证占比） | "收入结构"、"付费点" |
| 付费健康度 | 付费是选择还是强迫、资产是否保护、上限是否合理 | "氪金体验"、"商业化健康" |

**说明：** 本 skill 使用的 merged 数据为玩家评价与论坛帖子，**不包含真实付费埋点指标**。若仅基于 UGC，报告中必须注明：「本报告基于玩家评价与讨论归纳付费相关洞察，非来自真实付费数据。」若用户提供了指标文件，则按实际列进行指标级分析并注明数据来源。

---

## 二、从 UGC 识别付费/商业化信号

### 显式信号（直接提及付费）

- 评价/帖子中出现：氪金、充钱、付费、抽卡、保底、定价、太贵、不值、首充、礼包、通行证、Battle Pass、皮肤、直购、pay to win、gacha、货币、钻石、充值
- 与体验结合：不充玩不了、必须氪、付费墙、免费玩家体验差、付费即胜利

### 隐式信号（可推断为付费相关）

- **定价与价值感知**：贵、便宜、性价比、值不值、割韭菜
- **付费强迫感**：不充没法玩、逼氪、强迫付费
- **资产保护**：贬值、以前买的没用了、版本更新后废了、背刺
- **抽卡/概率**：概率不透明、保底太深、歪了、沉没成本

归类时与 `game_report` 的 **ops_knowledge.md** 中「问题归因决策树」保持一致（技术/商业化/运营/策划四层）；商业化层为本 skill 重点。

## 三、品类维度与付费分析重点

不同品类在分析付费时侧重点不同，下表供选择分析视角时使用（可与用户指定的 `genre` 或用户数据中的「品类」列结合）：

| 品类 | 核心玩家诉求 | 付费分析重点 | 典型风险 |
|------|-------------|-------------|---------|
| **MOBA** | 竞技公平、平衡感 | 外观付费占比、付费是否影响公平感知 | 付费即胜利感知、皮肤定价与价值 |
| **MMO** | 成长感、社交归属 | 订阅/月卡 vs 道具、大 R 与生态平衡 | 付费差距导致服务器失衡 |
| **二次元** | 角色情感、收藏满足 | 抽卡概率、保底、角色保值、情感价值 | 概率风波、老角色废弃引发信任崩盘 |
| **SLG** | 策略、联盟、霸主地位 | 付费差距、大 R 占比、免费玩家生存空间 | 顶级付费碾压 → 免费玩家流失 → 服崩 |
| **射击** | 竞技公平、操作感 | 外观/不影响数值的付费、Battle Pass | 付费影响公平会直接崩盘 |
| **卡牌** | 策略多样性、构筑乐趣 | 卡包/抽卡节奏、旧卡贬值速度 | 每版必买、旧卡快速废弃 → 付费压力过大 |
| **休闲/模拟** | 轻松感、创意、社交 | 付费墙是否阻断核心玩法、内购深度 | 内容重复 + 付费墙阻断玩法 → 口碑崩 |

若用户未指定品类，报告中使用「通用商业化框架」并注明未指定品类；若指定品类，则在结论与建议中紧扣该品类的「付费分析重点」与「典型风险」。

## 四、商业化健康判断三问

评估付费设计是否健康时，必须回答（见 AGENTS.md 商业化健康判断框架）：

1. **付费是选择还是强迫？** 免费玩家能否完整体验核心玩法？付费是锦上添花还是雪中送炭？
2. **历史资产是否被保护？** 已付费的皮肤/角色/装备会不会因为新版本更新而贬值？
3. **付费上限是否合理？** 获得完整体验需要的最大投入是多少？是否在目标用户承受范围内？

报告中「健康度评估」section 应基于 UGC 或用户数据对以上三点做定性或定量结论，并标注数据来源（UGC 或指标文件）。

## 五、付费模型参考表

| 付费模型 | 天花板高度 | 玩家留存影响 | 口碑风险 |
|---------|-----------|------------|---------|
| **Battle Pass** | 低（订阅制，固定费用） | 强（锁定留存行为） | 低（可预期、透明） |
| **Gacha（抽卡）** | 极高 | 中（依赖内容质量） | 高（概率不透明、保底设计影响口碑） |
| **直购道具** | 中 | 弱（无留存绑定） | 低（价值透明） |
| **赛季通行证** | 低 | 强（时限性驱动完成行为） | 低 |
| **氪金加速** | 高 | 中（付费差距影响公平感知） | 中到高（取决于差距幅度） |

分析时可结合游戏实际付费点与上表做对照，判断当前模型组合及风险区间。

## 六、用户提供数据的 Schema 建议

当用户上传或粘贴付费相关数据时，以下列名可作为识别与解析依据（均为可选，按实际存在列分析）：

- **维度**：`日期`、`游戏名`、`渠道`、`品类`、`区服`
- **指标**：`付费率`、`ARPU`、`ARPPU`、`LTV`、`首充率`、`首充时长`（或等价英文：pay_rate、ARPU、ARPPU、LTV、first_pay_rate、time_to_first_pay）

支持多游戏、多日期、多渠道；若存在 `品类` 列，可自动用于品类对比或按品类切片分析。若列名与上述不完全一致，可做合理映射（如「收入/用户」→ ARPU）并在报告中说明。

---

## 七、输出一致性

- **术语**：报告正文中统一使用「付费率」「ARPU」「ARPPU」「LTV」「首充」「付费结构」「健康度」等，与本文档一致。
- **归因**：若涉及问题归因，统一使用「技术层/商业化层/运营层/策划层」，与 game_report、game_retention 可互相引用。
- **数据边界**：在报告开头或「数据与局限」小节明确注明数据来源（仅 UGC / 仅用户提供指标 / 两者结合），以及是否存在真实付费埋点。
