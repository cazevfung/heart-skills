# 留存分析框架

本文件供 `game_retention` skill 的 Phase 3 使用。在从 UGC 归纳留存风险与建议时，按以下维度和术语保持一致性。

---

## 一、留存相关术语（与用户/报告用语对齐）

| 术语 | 含义 | 常见用户表述 |
|------|------|-------------|
| 次日留存 / D1 | 注册/安装后第 2 天仍活跃的比例 | "玩一天就删了"、"第二天就不想开了" |
| 7 日留存 / D7 | 第 7 天仍活跃 | "玩了一周就弃了"、"一周后没动力" |
| 30 日留存 / D30 | 第 30 天仍活跃 | "玩了一个月就退了" |
| 流失 / Churn | 用户不再回来 | "弃坑"、"退游"、"卸载"、"不玩了"、"afk" |
| 留存风险因素 | 可能导致流失的问题 | 从差评、论坛帖中归纳的「为什么有人会走」 |
| 留存改进建议 | 可执行的产品/运营动作 | 基于风险因素推导的优先级建议 |

**说明：** 本 skill 使用的数据为玩家评价与论坛帖子，**不包含真实 D1/D7/D30 指标**。报告中必须注明：「本报告基于玩家评价与讨论归纳留存风险与建议，非来自真实留存数据。」

---

## 二、从 UGC 识别留存/流失信号

### 显式信号（直接提及）

- 评价/帖子中出现：弃坑、退游、卸载、不玩了、玩不下去、玩几天就删、第二天没开、一周就腻、一个月就退了、afk、quit、uninstalled、dropped、left the game
- 与时间结合：玩 X 天/周/月 就……

### 隐式信号（可推断为流失风险）

- **技术类：** 崩溃、闪退、卡顿、进不去、更新后玩不了、设备不支持 → 易导致首日/次日流失
- **商业化类：** 太贵、不充玩不了、抽卡劝退、付费墙、贬值感 → 影响付费意愿与长期留存
- **运营类：** 太肝、活动劝退、沟通差、承诺未兑现、长草期太长 → 影响持续活跃与回流
- **策划类：** 无聊、内容少、平衡差、重复感强、没有目标 → 影响 7 日/30 日留存

归类时与 `game_report` 的 **ops_knowledge.md** 中「问题归因决策树」保持一致（技术/商业化/运营/策划四层）。

---

## 三、留存风险归因分层

对每条归纳出的「留存风险因素」，标注所属层级（可多选）：

| 层级 | 典型风险因素 | 更易影响的留存阶段（经验性） |
|------|-------------|---------------------------|
| **技术层** | 崩溃、闪退、卡顿、兼容性、登录失败 | 首日、次日 |
| **商业化层** | 付费强迫感、定价、抽卡体验、资产贬值 | 7 日、30 日及付费转化 |
| **运营层** | 肝度、活动设计、沟通、长草、补偿 | 7 日、30 日、回流 |
| **策划层** | 玩法深度、内容量、平衡、目标感、重复感 | 7 日、30 日 |

归纳时注明：**现象描述**、**出现频次/占比（若可统计）**、**引用来源（评价/帖子摘要或编号）**。

---

## 四、报告中的建议结构

每条「留存改进建议」建议包含：

1. **对应风险**：来自上述哪一条或哪一类风险因素
2. **建议动作**：具体、可执行（如「在登录流程增加 shaders 预编译与进度条」而非「优化体验」）
3. **预期影响**：对哪类用户、对 D1/D7/D30 的预期方向（定性即可，如「有助于提升次日留存」）
4. **优先级**：P0/P1/P2 或 高/中/低（可与 game_report 的优先级用语统一）
5. **参考**：若有同类产品做法或行业惯例，可简要标注

---

## 五、与运营阶段结合（可选）

若已读取 `game_report` 的 **ops_knowledge.md**，可先判断当前运营阶段（买量冲刺期、版本发布期、内容消化期等），再在留存报告中简要说明：

- 当前阶段下，**最应优先关注的留存问题**（例如版本发布期优先看首日/次日技术问题；版本末尾疲劳期优先看内容与目标感）
- 避免建议与阶段错位（例如在危机响应期不建议优先做长期内容规划，而应先稳口碑与信任）

---

## 六、输出一致性

- **术语**：报告正文中统一使用「次日留存」「7 日留存」「30 日留存」「流失」「留存风险因素」「留存改进建议」
- **归因**：统一使用「技术层/商业化层/运营层/策划层」，与 game_report 的归因结论可互相引用
- **数据边界**：在报告开头或「数据与局限」小节再次强调：本报告基于玩家 UGC 归纳，非真实留存数据
