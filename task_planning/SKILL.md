# Subagent Task Planning Skill

## 核心原则

**先理解，再拆解；先验证，再执行；我是整合者，不是传声筒。**

---

## 第一部分：深度理解需求

### 1.1 需求澄清三问

面对任何任务请求，先问自己：

**Q1: 最终要交付什么？**
- [ ] 一个结论？（是/否、排名、趋势判断）
- [ ] 一份材料？（报告、文案、图片、数据文件）
- [ ] 一个决策依据？（对比分析、风险评估、机会识别）
- [ ] 一个执行方案？（步骤清单、资源配置、时间表）

**Q2: 交付物的"质量"由什么决定？**
- [ ] 数据覆盖度？（多少平台、多少样本、多长时间跨度）
- [ ] 信息时效性？（最新数据 vs 历史数据）
- [ ] 分析深度？（现象描述 vs 根因分析 vs 预测建议）
- [ ] 呈现形式？（原始数据 vs 结构化报告 vs 可视化图表）

**Q3: 用户没说但隐含的需求？**
- [ ] "分析一下" → 隐含：要有结论，不是罗列
- [ ] "详细" → 隐含：结构化、有数据支撑
- [ ] "竞品" → 隐含：要有对比、要有差异化洞察
- [ ] "报告" → 隐含：可阅读、可分享、有 executive summary

### 1.2 识别信息缺口

列出完成任务需要的所有信息元素：

|---------|---------|---------|--------|

**P0**: 没有就不能完成任务  
**P1**: 有的话质量显著提升

### 2.2 Agent 选择决策树

```
任务需要外部数据？
  ├─ 是 → 需要 game-crawler
  │       └─ 数据量很大/很复杂？
  │           ├─ 是 → 拆分为多个并行抓取任务
  │           └─ 否 → 单个抓取任务
  └─ 否 → 不需要 game-crawler

任务需要创意内容？
  ├─ 是 → 需要 xiaoyizong
  │       └─ 需要基于数据创作？
  │           ├─ 是 → 先 game-crawler 抓取 → 再 xiaoyizong 创作
  │           └─ 否 → 直接 xiaoyizong 创作
  └─ 否 → 不需要 xiaoyizong

任务需要专业分析？
  ├─ 是 → main（我）来做
  │       └─ 需要基于外部数据？
  │           ├─ 是 → 先召唤 game-crawler → 我分析
  │           └─ 否 → 我直接分析已有信息
  └─ 否 → 可能不需要 main 深度参与
```

### 2.3 常见任务类型的标准拆解

#### 类型 A: 竞品分析报告

**典型需求**: "分析一下XX的竞品情况"

**信息元素清单**:
- P0: 本品信息（定位、功能、用户评价）
- P0: 竞品列表（谁是竞品？3-5个）
- P0: 竞品基本信息（定位、功能）
- P0: **视频平台用户反馈（B站/YouTube）⚠️ 易遗漏**
- P1: 应用商店评价对比（本品 vs 竞品）
- P1: 社区讨论（Reddit/知乎等）
- P1: 运营策略观察（社交媒体、活动）
- P2: 市场数据（下载量、收入估算）

**标准拆解**:
```
Step 1: main 确定竞品范围
  └─ 输出：竞品列表（3-5个）
  └─ 确定数据源（必须包含B站/YouTube视频）

Step 2: 并行抓取（game-crawler × N）
  ├─ Subtask 2a: 抓取本品应用商店信息
  ├─ Subtask 2b: 抓取竞品A应用商店信息
  ├─ Subtask 2c: 抓取竞品B应用商店信息
  ├─ Subtask 2d: 抓取竞品C应用商店信息
  ├─ Subtask 2e: **抓取B站相关视频和评论 ⚠️**
  └─ Subtask 2f: **抓取YouTube相关视频和评论 ⚠️**
  └─ 每个子任务输出：结构化数据文件

Step 3: main 验证数据完整性
  └─ 检查：文件是否存在、格式是否正确、数据是否充足
  └─ **重点检查：视频平台数据是否抓取**
  └─ 如有缺失：补充抓取或调整分析范围

Step 4: main 整合分析
  ├─ 读取所有数据（含视频平台评论）
  ├─ 对比分析（功能、定位、用户评价）
  ├─ **视频平台用户反馈专项分析**
  ├─ SWOT分析
  └─ 生成报告

Step 5: main 输出成果
  └─ 输出：竞品分析报告（markdown/docx/飞书）
```

#### 类型 B: 游戏舆情分析

**典型需求**: "分析一下XX游戏的口碑"

**信息元素清单**:
- P0: 应用商店评分和评论
- P0: 社区讨论（TapTap/Reddit等）
- P0: **视频平台讨论（B站/YouTube）⚠️ 必做**
- P1: 近期版本更新内容
- P1: 官方公告和回应
- P2: 直播/短视频平台（抖音/TikTok）

**标准拆解**:
```
Step 1: game-crawler 抓取多平台数据
  ├─ Subtask 1a: 抓取应用商店评论
  ├─ Subtask 1b: 抓取TapTap论坛
  ├─ Subtask 1c: 抓取Reddit讨论
  ├─ Subtask 1d: **抓取B站相关视频评论 ⚠️**
  └─ Subtask 1e: **抓取YouTube相关视频评论 ⚠️**

Step 2: main 验证数据质量
  └─ 检查：评论数量、时间分布、关键词覆盖
  └─ **重点检查：视频平台数据是否充足**

Step 3: main 分析
  ├─ 情感分析（正面/负面比例）
  ├─ 主题聚类（玩家关注什么）
  ├─ **视频平台专项分析（真实体验反馈）**
  ├─ 趋势观察（近期变化）
  └─ 根因分析（问题归因）

Step 4: main 输出报告
```

#### 类型 C: 公告文案+配图

**典型需求**: "写一个XX活动的公告，要配图"

**信息元素清单**:
- P0: 活动主题和内容
- P1: 游戏风格和调性（需要参考）
- P1: 目标玩家群体
- P2: 竞品类似活动参考

**标准拆解**:
```
Step 1: main 确认需求细节
  └─ 活动主题、时间、奖励、目标平台

Step 2: （可选）game-crawler 抓取参考
  └─ 如果不知道游戏风格，抓取官方历史公告作为参考

Step 3: xiaoyizong 创作文案
  └─ 输出：公告文案

Step 4: xiaoyizong 生成配图
  └─ 输出：公告配图（横版+竖版）
  └─ 超时设置：10分钟（生图较慢）

Step 5: main 审核整合
  ├─ 检查文案是否符合调性
  ├─ 检查图片质量
  └─ 输出：完整公告包（文案+配图）
```

#### 类型 D: 游戏数据抓取

**典型需求**: "抓一下XX游戏的数据"

**标准拆解**:
```
Step 1: main 确认抓取范围
  └─ 平台（TapTap/Reddit/YouTube...）
  └─ 数据类型（评论/帖子/视频...）
  └─ 时间范围
  └─ 数量要求

Step 2: game-crawler 执行抓取
  └─ 可能拆分为多个并行任务（多平台）

Step 3: main 验证数据
  └─ 检查：文件大小、格式、字段完整性
  └─ 抽样检查数据质量

Step 4: main 输出
  └─ 原始数据文件路径
  └─ 数据摘要（数量、时间分布等）
```

## 第三部分：任务拆解执行手册

### 3.1 拆解检查清单

每个子任务必须明确：

- [ ] **输入**: 上一步的输出 / 已知信息 / 需要抓取的数据源
- [ ] **输出**: 具体的文件/数据/结论，格式是什么
- [ ] **负责 Agent**: 谁能做这个任务（参考能力矩阵）
- [ ] **完成标准**: 怎么算做完了（文件存在？数据量达标？）
- [ ] **超时预估**: 预计多久完成（决定等待策略）
- [ ] **依赖关系**: 必须等哪个任务完成后才能开始

### 3.2 依赖关系设计

**并行执行**（无依赖）：
```
├─ Task A: game-crawler 抓取平台1
├─ Task B: game-crawler 抓取平台2
└─ Task C: game-crawler 抓取平台3
```

**串行执行**（有依赖）：
```
Task 1: game-crawler 抓取数据
  ↓ [验证输出]
Task 2: main 分析数据
  ↓ [验证结论]
Task 3: xiaoyizong 生成配图（基于分析结论）
```

**混合模式**（常见）：
```
Phase 1: 并行抓取
  ├─ Task 1a: 抓取A
  └─ Task 1b: 抓取B
  ↓ [全部完成后]
Phase 2: 分析整合
  └─ Task 2: main 分析
  ↓
Phase 3: 创作输出
  └─ Task 3: xiaoyizong 生成内容
```

### 3.3 检查点设置

**每个 Phase 结束后问自己：**

| 检查点 | 问题 | 如果No怎么办 |
|-------|------|------------|
| 数据获取后 | 数据质量够吗？（数量、时效、完整性） | 补充抓取 / 调整分析范围 |
| 数据分析后 | 结论有数据支撑吗？ | 补充分析 / 降低结论强度 |
| 内容生成后 | 符合需求吗？（风格、调性、信息准确） | 修改需求 / 重新生成 |
| 最终交付前 | 能回答用户原始问题吗？ | 补充缺失部分 |

### 3.4 超时与异常处理

| 场景 | 处理策略 |
|-----|---------|
| Subagent 超时 | 检查是否有 checkpoint 文件 → 询问是否继续等待 / 接受部分结果 / 放弃 |
| 数据质量不达标 | 明确告知用户数据缺口 → 询问是否继续用现有数据 / 补充抓取 / 调整任务范围 |
| Subagent 返回不完整 | 追问缺失部分 / 拆解为更细子任务重新执行 |
| 发现新需求 | 评估是否在当前任务内 / 另开新任务 |

```

  │   ├─ 有 → 询问用户：继续等待 / 接受部分结果 / 放弃

  ├─ 提供选项：
  │   ├─ A: 用现有数据继续（降低结论强度）

  │   └─ C: 调整任务范围（聚焦能拿到的数据）

  ├─ 追问缺失部分
  ├─ 如果无法补全 → 拆解为更细子任务
  └─ 如果多次失败 → 考虑换 agent 或人工介入

发现新需求（超出原范围）？
  ├─ 评估复杂度
  │   ├─ 简单 → 纳入当前任务
  │   └─ 复杂 → 另开新任务，告知用户
  └─ 不擅自扩大范围
```

## 附录：常见错误与纠正

### 错误 1: 把分析任务直接扔给 game-crawler
**表现**: "你分析一下这个游戏的口碑" → 给 game-crawler  
**问题**: game-crawler 只会抓数据，不会分析  
**纠正**: game-crawler 抓数据 → main 分析 → 输出报告

### 错误 2: 没有设置检查点
**表现**: 一次性给 subagent 一个大任务，等待 10 分钟，结果质量差  
**问题**: 无法中途调整，返工成本高  
**纠正**: 拆分为小步骤，每步验证后再继续

### 错误 3: 放弃整合责任
**表现**: subagent 给什么就转发什么，不做筛选和整合  
**问题**: 用户得到的是原始材料，不是有价值的成果  
**纠正**: 我是最终整合者，必须对输出质量负责

### 错误 4: 忽视 agent 能力边界
**表现**: 让 xiaoyizong 去抓数据，让 game-crawler 写文案  
**问题**: 效率低，结果差  
**纠正**: 严格按照能力矩阵分配任务

### 错误 5: 没有明确输出格式
**表现**: "写个报告" → subagent 不知道要什么格式  
**问题**: 返回的内容不符合预期  
**纠正**: 明确指定：markdown/docx/飞书/图片/数据文件

## 第六部分：实战检查清单

### 6.1 任务启动前必做

