# Game Report Skill

本 skill 是游戏分析的**展示层**。它从 `game_crawl` skill 的持久化数据存储中读取数据，套用模板注册表中对应的报告模板，生成结构化分析报告。本 skill 不直接爬取任何数据。

## 参数提取

从用户消息中提取以下参数。未提及的参数使用默认值。**执行前先 Echo 所有已解析参数。**

| 参数 | 含义 | 默认值 |
|------|------|--------|
| **game** | 游戏名称（中文或英文均可） | 从用户消息推断，无法推断时询问用户 |
| **report_type** | 报告类型（见模板注册表） | 从意图推断；无法推断时询问用户 |
| **version** | 版本号，仅 `version` 类型时使用 | 最新版本 |
| **platform** | 数据来源平台（单个平台名或 `all`） | `all`（该游戏所有已配置平台） |
| **date_range** | 评价/帖子时间范围 | 近3个月 |
| **include_competitor** | 是否包含竞品对比（仅综合报告） | `yes` |
| **report_format** | `markdown` / `feishu` | `feishu` |
| **report_language** | 报告语言 | 与用户提问语言一致 |
| **output_path** | 输出目录路径 | `user/documents/游戏分析报告/` |
| **output_destination** | 报告输出位置：`file` 仅写文件，`chat` 仅在对话中输出完整报告 md | `file` |

## 参考模板（可选）

以下模板作为**参考素材库**，在不确定某类内容如何组织时可查阅。报告结构不由模板决定，而由 Phase 3.0 的自我追问决定。

| report_type | 适用场景 | 所需 data_types | 参考模板 | 文件名后缀 |
|-------------|---------|----------------|---------|-----------|
| sentiment | 口碑风险扫描、玩家情绪分析 | `app_info`, `reviews`, `forum_posts`, `videos`(可选) | `template_sentiment.md` | 舆情分析报告 |
| version | 版本内容复盘、更新效果评估 | `changelogs`, `forum_posts`, `videos`(可选), `announcements`(可选) | `template_version.md` | 版本内容分析 |
| comprehensive | 综合诊断、竞品对比 | `app_info`, `reviews`, `forum_posts`, `changelogs`, `videos`(可选), `announcements`(可选) | `template_comprehensive.md` | 综合分析报告 |

**使用方式：** 当自我追问后仍不确定某 section 如何撰写时，可参考对应模板的写作方式，但不必遵循其结构。

## 意图推断规则

**当由工单（create_task 的 subtask）调用且参数已全部在工单中给定时**，跳过询问、直接使用工单中的 game、report_type 等参数。以下「立即询问」规则仅适用于**直接对话场景**（用户当面请求生成报告时）。

### 推断 game

如果用户消息中完全无法识别任何游戏名称，**立即询问**，不要继续执行：
> "你想分析哪款游戏？"

### 推断用户关切焦点

不急于确定 report_type，而是从用户消息中提取关键词，理解其潜在关切：

| 用户关键词 | 潜在关切 | 建议 data_types |
|-----------|---------|----------------|
| 口碑、评价、评分、差评、玩家反馈 | 当前口碑风险与情绪分布 | `app_info`, `reviews`, `forum_posts` |
| 版本、更新、patch、新内容、改动 | 版本更新效果与玩家反应 | `changelogs`, `reviews`, `forum_posts` |
| 竞品、对比、市场、格局 | 相对位置与竞争态势 | `app_info`, `reviews`（本品+竞品） |
| 留存、流失、活跃 | 玩家生命周期健康度 | `app_info`, `reviews`, `forum_posts` |
| 付费、氪金、商业化 | 变现设计与玩家接受度 | `reviews`, `forum_posts` |

如果用户消息过于模糊（例如仅说"分析一下王者荣耀"），**简要确认其关切焦点**：

> "你想重点了解这款游戏的哪个方面？
> - 当前口碑和玩家反馈
> - 最新版本的表现
> - 与竞品的对比
> - 整体运营健康度"

等待用户回复后再继续。

### 推断 output_destination

用户明确表示「在对话框看」「直接发聊天」「我要 chat」「发到对话」等时，将 `output_destination` 设为 `chat`。未提及则使用默认值 `file`。

## 执行流程：4 个阶段

### Phase 1 — 理解

**本 skill 作为后台 subtask 执行时，禁止在结果中向用户提问。** 当准入条件不满足（例如上游 st-0 成功平台数 &lt; 2）时，**直接以 failed 退出**，并在 result 中写出**结构化原因**（如 `reason: "upstream_platforms_insufficient"`, `success_platforms: 1`, `required: 2`），由对话 agent 根据该原因向用户提问并给出选项。

1. 解析用户消息或工单，填写参数表。
2. 推断 `game`（工单已给定时直接使用；直接对话且不明确时询问用户）。
3. **理解用户意图**：不急于确定 report_type，而是先理解用户问题的核心关切。从用户消息中提取关键词，判断其潜在关注点（口碑风险？版本表现？竞品对比？长期趋势？）。
4. 从模板注册表读取推荐的 `data_types` 列表（作为参考，非强制）。
5. Echo 所有已解析参数，包括推断的用户意图焦点。

### Phase 2 — 数据就绪检查

**准入条件检查（工单调用时）：** 当本 skill 作为 st-1 执行时，先读取 st-0 的采集摘要。若摘要显示「成功平台数 &lt; 2」或「最低完成标准未满足」（如 `minimum_met: false`）：**不询问用户**，本 subtask 标记为 **failed**，在 result 中输出结构化原因（如 `reason: "upstream_platforms_insufficient"`, `success_platforms: 1`, `required: 2`），然后结束。仅当准入条件已满足（成功平台数 ≥ 2）时，才继续下面的数据就绪检查与报告生成。「在报告开头注明数据覆盖范围」仅用于准入已满足但部分平台缺失时的说明，不用于「是否继续执行」的决策。

**数据根目录：** 固定为项目根目录下的 `data/game_data`

**确定要检查的平台范围：**
- `platform = all`（默认）：调用 **read_game_registry**，从返回的注册表中取该游戏的所有已配置平台。
- `platform = <具体平台>`：仅检查该平台。

**对每个 `(平台, 数据类型)` 组合进行检查：**
从 `game_crawl` 平台注册表确定该平台提供哪些 data_types，与本 `report_type` 所需的 data_types 取交集。对交集中的每个组合检查：

```
merged/<平台>/<game_id>/<data_type>/data.json
```

判断规则：
- 文件存在且 `last_updated` 在 **24 小时内** → 就绪，直接读取
- 文件不存在，或 `last_updated` 超过 24 小时 → 标记为"需要爬取"
- **可选 data_type**（如 `videos`, `announcements`）：
  - 先调用 **read_game_registry**，查该游戏是否已配置对应平台
    （`videos` → 检查 `youtube` / `bilibili`；`announcements` / browser-changelogs → 检查 `official_site`）
  - **已配置** → 按上述规则正常判断（文件缺失或过期 → 标记为需要爬取）
  - **未配置** → 跳过，不标记为需要爬取
- 用户明确说"使用现有数据" → 跳过爬取，即使数据过期

**如果有任何组合需要爬取：**
委托 **game_crawl skill**，传入：
- `game` = 当前游戏
- `data_types` = 需要更新的 data_type 列表
  （**必须同时包含必需 data_type 和所有"已配置平台的可选 data_type"**，
  不得仅传 required data_types；如无特殊限制，建议直接传 `auto`）
- `platforms` = 涉及的平台列表（建议传 `auto`，让 crawl_runner 自动匹配所有已配置平台）

等待 game_crawl 完成后继续。

如果 game_crawl 对某个 data_type 失败，在最终报告的对应 section 标注"数据不足"，继续生成其他 section。

### Phase 3 — 分析与报告生成

#### Phase 3.0 — 自我追问框架（核心思维流程）

**本阶段是报告质量的关键。不要急于套用模板，而是通过自我追问让洞察自然生长。**

数据加载完成后，按以下框架进行自我追问。每个问题的答案将决定报告的结构和内容重点。

**第一层追问：表象识别**
- 数据呈现了什么样的表面现象？（评分高低、情绪分布、话题热度）
- 这些现象是否符合预期？如果不符合，偏差在哪里？
- 时间维度上，这是突发变化还是渐进趋势？

**第二层追问：矛盾挖掘**
- 数据中是否存在相互矛盾的信号？（例如评分高但情绪负面、讨论热但评价少）
- 不同平台/群体的反馈是否一致？差异意味着什么？
- 官方说法与玩家感知之间有无落差？

**第三层追问：归因定位**
- 问题的根因属于哪个层面？（技术/商业化/运营/策划）
- 这是设计缺陷、执行失误、沟通问题还是时机不当？
- 责任方是谁？解决路径是什么？

**第四层追问：演变推演**
- 如果不干预，这个问题会如何发展？
- 下版本发布时，当前问题会被放大还是自然消退？
- 什么条件下问题会转化为危机？

**第五层追问：机会识别**
- 数据中是否隐藏着被忽视的机会点？
- 玩家的哪些诉求尚未被满足但可被满足？
- 竞品是否也在面临同样问题？我们的相对位置如何？

**追问输出**：将以上追问的答案整理为「核心发现清单」，作为后续 section 的写作基础。

#### Phase 3.0 — 运营上下文评估（强制步骤，在所有 section 生成前执行）

读取 `references/ops_knowledge.md`，完成以下评估。评估结果不直接输出为独立段落，而是作为隐含上下文贯穿后续所有 section 的叙述，影响每个问题的严重程度判断、建议优先级和改进方案选择。

**步骤 3.0.1 — 识别当前运营阶段**

对照 `ops_knowledge.md` 第 1 节"运营阶段识别"中的信号对照表，根据：
- 评价时间分布（近 24-48 小时评价量 vs 日均）
- 论坛帖子主题分布
- 最后版本更新距今天数
- 竞品提及频率变化

判断当前处于哪个阶段：买量冲刺期 / 版本发布期 / 内容消化期 / 版本末尾疲劳期 / 版本空档期 / 危机响应期

**步骤 3.0.2 — 对核心负面问题运行归因四分法**

对排名前 5 的负面问题，各判断其归属层（可多层）：
- **技术层**：Bug、崩溃、性能、匹配系统质量
- **商业化层**：定价、付费强迫感、历史资产保护
- **运营层**：活动设计、沟通方式、版本节奏、预期管理
- **策划层**：玩法设计、平衡性、内容深度、成长曲线

对照 `ops_knowledge.md` 第 2 节"问题归因决策树"进行判断。

**步骤 3.0.3 — 时间维度风险评估**

对步骤 3.0.2 中归因后的每个核心问题，判断：
- 该问题在下版本是否有恶化风险（参考 `ops_knowledge.md` 第 3 节"时间维度风险评估"）
- 处理时限：立即 / 本版本内 / 下版本前

**步骤 3.0.4 — 将以上评估写入分析上下文**

在内部形成如下分析上下文（不需要直接输出，但影响所有后续 section 的叙述质量）：
```
当前运营阶段：[阶段名称] — [2句依据]
核心问题归因：
  - [问题1]：[归因层] — [下版本风险🔴/🟡/🟢]
  - [问题2]：[归因层] — [下版本风险]
  ...
```

#### Phase 3.1 — 多平台数据聚合加载与报告生成

**数据加载：** 对每个所需 `data_type`，从所有相关平台的 merged 目录加载并合并。调用 **read_game_data** 时，为控制上下文长度、避免 API 报错，**必须传入抽样参数**：`max_items=50`（或按需 30～80）；若单次返回仍过长，可再加 `max_output_chars=80000`。返回中的 **`_meta`**（若有）表示抽样信息（`total_count`、`returned_count`），报告中应简要说明抽样范围（如「基于 N 条数据，抽样 M 条分析」）。

   - `app_info` → 读取 `info` 对象（通常来自 taptap，如有多个平台取最新的）；无列表，可不传 max_items
   - `reviews` → 读取 `reviews` 数组，按 `date_range` 过滤 `created_at`
   - `forum_posts` → 从 taptap 和 reddit 分别加载 `posts` 数组，合并后按 `created_at` 统一排序，按 `date_range` 过滤
   - `changelogs` → 从 taptap 和 official_site 分别加载 `versions` 数组，按 `version` 字符串去重合并（同一版本取 body 更详细的条目），按 `release_date` 降序排列；如指定 `version`，过滤匹配的条目
   - `videos` → 从 youtube 加载 `videos` 数组，按 `date_range` 过滤 `published_at`
   - `announcements` → 从 official_site 加载 `announcements` 数组，按 `date_range` 过滤 `published_at`

**数据来源填写规则：** 报告头部的「**数据来源**」必须根据**本次实际成功加载的平台**填写。先根据 read_game_registry 与本次 report_type 所需 data_types 确定要检查的平台列表；对每个平台尝试从 merged 加载。仅当某平台已纳入本次检查范围且对该平台的加载失败或对应数据为空时，方可在「数据覆盖范围」中注明该平台「不可用」或「数据为空」。成功且有内容的平台一律列入「本报告基于 …」，不得对本次已成功取得数据的平台写为不可用。

#### Phase 3.1.0 — 引用索引构建（强制，生成任何正文前执行）

在数据加载完毕后、按模板生成报告正文之前，必须执行本步骤。

1. **读取** `references/citation_instructions.md`，熟悉引用格式规范。
2. **扫描** 所有已加载数据（`reviews`、`forum_posts`、`videos`、`changelogs`、`announcements`）。
3. **选出** 计划在报告中具体引用的条目（典型引用、支撑热度/数字的帖子或评价、痛点与好评的示例）。
4. **分配编号**：从 1 开始逐条分配唯一编号，在内部构建 citationMap（编号 → 条目元数据：标题/摘要、平台、日期、url）。
5. **贯穿写作**：生成后续每个 section 时，凡引用具体数据或引文，必须在句末或引文末加上对应 [N]；报告写完后，在文件末尾输出 `## 引用来源` 章节（格式见 `citation_instructions.md`）。

**引用密度要求：**
- 每个 blockquote 引文（`> "引文"`）必须附 [N] 标注。
- 每个痛点或好评中「提及 X 条」等数字，必须附 [N] 以追溯到支撑该数字的帖子/评价。
- 每个「典型反馈」「代表性评价」必须附 [N]，对应到具体一条 review 或 post。

#### Phase 3.1.1 — 报告结构原则与 Section 生成

**核心原则：让问题自己长出结构**

报告结构不由模板预先决定，而由 Phase 3.0 自我追问的「核心发现清单」决定。按以下原则组织内容：

**结构决策流程：**
1. **开篇结论先行** — 用一段完整文字回答：核心发现是什么？主要风险/机会在哪里？
2. **按逻辑递进组织主体** — 根据核心发现的内在逻辑决定 section 顺序：
   - 如果是「问题诊断型」：现象 → 归因 → 推演 → 建议
   - 如果是「趋势观察型」：历史 → 现状 → 预测 → 应对
   - 如果是「对比分析型」：本品现状 → 竞品对照 → 差距分析 → 机会识别
3. **数据支撑后置** — 详细数据表格、引用来源放在对应洞察之后，不干扰主线阅读
4. **附录收尾** — 方法说明、完整数据、引用列表

**Section 动态生成规则：**
- 不是从模板选 section，而是根据「核心发现清单」判断需要哪些 section
- 每个 section 必须有明确的「存在理由」——它回答了什么追问？支撑了什么洞察？
- section 标题可以自定义，不必套用模板标题

**模板参考（降级为可选）：**
`references/template_*.md` 文件作为「参考素材库」而非「结构模板」。当不确定某类内容如何组织时，可参考对应模板的写作方式，但不必遵循其结构。

**Section 索引（强制）**：在报告头部（或紧接开篇结论之后）必须提供**「本报告 section 索引」**（目录），列出本报告所有一级 section 的标题及出现顺序。索引应反映本次分析的内在逻辑，而非固定模板顺序。

**撰写要求（贯穿所有 section）：**
   - 不要直接复制粘贴原始评论，要提炼分析观点
   - 典型引用可直接引用（用引号或 blockquote 格式），**且必须附 [N] 标注**（见 Phase 3.1.0 与 `citation_instructions.md`）
   - 严重程度、热度等评估基于 `score`（点赞数）和提及频次
   - 聚合多平台数据时，注明数据来源平台和区域（海外/国服），必要时分别分析
   - **运营上下文注入：** 生成每个 section 时，将 Phase 3.0 的评估结果用于：
     - 解读每个痛点的严重程度时，说明其归因层（是策划/运营/商业化/技术哪层的问题）
     - 给出改进建议时，优先级排序结合时间维度风险评估
     - "总结与建议" section 中，必须包含"运营诊断"内容（参见模板）

如 `include_competitor=yes` 且模板包含竞品对比 section：从 game_crawl 数据中读取竞品数据（如有），或委托 game_crawl 爬取主要竞品的 `app_info`。

**段落深度强制规则（不受 Token 节省规则约束）：**
0. **开篇结论（强制）：** 开篇结论须为一段连贯的大段落（建议 8–15 句或约 200–400 字），仅文字、不包含表格或列表。**结论先行**：先给出核心结论、关键发现与主要风险/机会判断，再以简要依据支撑，而非先铺数据再总结。**高洞察**：须隐含或显式回答（1）本质问题是什么（2）主要原因（3）若不干预下阶段如何演变；禁止仅复述表格或罗列数字。生成时可利用 Phase 3.0 的运营阶段与归因结论。
1. **洞察优先于描述**：段落中禁止出现复述数字的句子，必须将数据转化为判断。错误示例："负面评价占比为 42%"；正确示例："42% 的负面占比已进入关注区间，结合评分持续下滑的信号，表明问题正在从局部投诉演变为系统性口碑压力"。
2. **三问框架**：每个含表格/要点的 section 后的解读段落，必须隐含回答：（1）这组数据的**本质问题**是什么（不是表面现象）？（2）是**什么原因**造成了这个结果？（3）如果不干预，**下阶段如何演变**？
3. **痛点归因显式标注**：每个负面痛点后必须注明归因层（策划/运营/商业化/技术），并给出 1 句理由。
4. **建议可执行标准**：P0/P1 建议必须写明至少 2 个具体执行步骤，禁止出现"优化 X 体验"之类的模糊表述。
5. **后半部分质量**：报告后半部分（玩家分层、热点话题、运营诊断、竞品对比、总结与建议）与前半部分同样重要，每节都须包含至少 2–3 段连贯深度洞察段落，不得在后期章节缩水为仅要点。
6. **禁止报告内使用 emoji**：报告正文中不得使用任何 emoji（如 🟢🟡🔴★☆）；所有等级、程度、热度等均用文字表述（正常/关注/警戒、高/中/低、极高/高/中/低等），不得使用符号替代。

**输出前自检（Phase 4 执行前）：** 在写最终报告或调用 feishu 前，自检：除表格与要点外，每个一级 section 是否都有一段以上 3–5 句的纯文字深度解读段落；开篇是否有单独大段落结论；若某 section 仅有要点/表格而无段落，必须补写后再输出。

## Markdown 表达决策树

**根据洞察类型选择最合适的表达格式**

| 洞察类型 | 推荐格式 | 使用场景 |
|---------|---------|---------|
| **时间因果链** | 时间线 / 流程图 | 展示事件先后与因果关系，如"版本发布后口碑如何演变" |
| **多方对比** | 对比表格 | 本品vs竞品、现状vs历史、不同平台差异 |
| **归因拆解** | 层级列表 / 树状结构 | 逐层拆解问题归属（技术/运营/商业化/策划） |
| **情绪证据** | 引用块 + 来源标注 | 玩家原声、典型反馈，增强可信度 |
| **反事实推演** | 折叠块对比 | "如果当时...现在会怎样"，主路径清晰，细节可展开 |
| **矛盾/异常发现** | 高亮块 + 解释 | 强制注意力，如"评分高但情绪负面"的gap |
| **数据支撑细节** | 折叠表格 / 附录 | 不干扰主线，需要时可展开 |
| **复杂关系网络** | 矩阵图 / 交叉表格 | 多维度交叉分析，如"玩家分层 × 诉求类型" |

**格式选择原则：**
1. **优先服务洞察** — 格式是让洞察更易被理解，不是为了美观
2. **密度控制** — 关键发现用高亮，支撑细节用折叠，避免信息过载
3. **阅读节奏** — 文字段落与表格交替，避免连续大段文字或连续表格
4. **引用规范** — 所有数据、引文必须标注来源 [N]，见 `citation_instructions.md`

### Phase 4 — 输出

**确定 game_id：** 调用 **read_game_registry**，从返回的注册表中读取游戏的各平台配置（TapTap app_id、reddit subreddit、YouTube channel 等），用于构建各平台的数据路径。如注册表中找不到该游戏，先委托 game_crawl 自动发现并注册。

**markdown 格式：** 按 `output_destination` 分支执行。

- **`file`**（默认）：
  - 文件名格式（从模板注册表读取后缀）：`{YYYY-MM-DD}_{游戏名}_{文件名后缀}.md`（例：`2026-03-05_王者荣耀_舆情分析报告.md`）
  - 写入 `output_path` 目录
  - 对话中必须打印：完整文件路径 + 报告摘要（3 句话：游戏、类型、核心结论）；并**必须**附带一条 Markdown 链接，便于用户在控制台「用户文档」中点击打开：`[在应用中打开报告](/user-docs?file=<relative_path>)`，其中 `<relative_path>` 为相对于 `user` 目录的路径（例如默认 output_path 下为 `documents/游戏分析报告/2026-03-05_王者荣耀_舆情分析报告.md`；若 output_path 为其他子目录，则 relative_path 与之对应，如 `documents/自定义子目录/文件名.md`）。
- **`chat`**：
  - 不写本地文件
  - 在回复中**直接输出完整报告正文**（完整 Markdown 内容），便于在对话里阅读或复制

**feishu 格式：**
1. 不写本地文件，在内存中保留完整报告 Markdown 正文。
2. 调用 **feishu_create_doc** 工具：
   - title = `{游戏名}游戏分析报告 — {YYYY-MM-DD}`
   - content = 上一步在内存中的完整报告 Markdown 正文
   - folder_token = 环境变量 FEISHU_FOLDER_TOKEN（若已配置）
3. 工具返回后，打印飞书文档 URL 和报告摘要（3 句话：游戏、类型、核心结论）

## 失败处理

- 若某个 data_type 的数据文件不存在且 game_crawl 也失败：在对应报告 section 标注"数据不足（data_type 爬取失败）"，继续生成其他 section
- 若所有 data_type 均不可用：告知用户，建议先运行 game_crawl 收集数据
- 若 read_game_registry 返回的注册表中无该游戏：委托 game_crawl 自动发现后继续

## Token 节省规则

- 只读取本次 report_type 所需的 data_type，不加载多余数据
- 只在 Phase 4 写最终报告文件（当 `output_destination=file`）；不保存中间分析文件
- 当 `output_destination=chat` 时，报告正文会出现在对话中，注意单条消息长度
- 竞品数据只需读取 `app_info`（评分、核心特色），不需要完整报告
- 委托 game_crawl 时只传入真正缺失的 data_type，避免不必要的重爬
- **分析深度不受 token 节省规则约束**：Section 4 的每个维度"**分析：**"段落、Section 7"改进建议"表格的"建议方案"列，均为必须完整输出的内容，在 subtask 模式下同样不得省略或缩减
